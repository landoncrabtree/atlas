## Ranked bottleneck list

Each entry names a **location**, a **category**, the **measured impact**
(from the baseline benches above or from static reasoning against the
principles in `.github/instructions/performance.instructions.md`), a
**fix hypothesis**, and a **confidence** rating.

### 1. `atlas_fs::view_model::Inner::recompute` — quadratic loader

- **Location**: `crates/atlas-fs/src/view_model.rs:93-102`, called from
  `run_loader` at line 238 on every `ListEvent::Batch`.
- **Category**: allocation + wasted CPU (worst case `O(N²)`).
- **Measured impact**: `open_and_load_10000` is **20.8×** slower than
  `open_and_load_1000` for 10× the data — a linear implementation
  would be ~10×. On the 10k bench the loader spends ~700 ms almost
  entirely inside `recompute`. This directly threatens the MVP goal
  "open a 100k-file directory and scroll smooth 60 fps": the pane
  never finishes loading. Additionally we clone every entry into a
  fresh `view` vec on every batch.
- **Fix hypothesis**: incrementally maintain `view`. When a batch of
  new entries arrives, run `matches` on each, sort just the new subset,
  then merge-in via `partition_point` or a single final `sort_by`.
  Alternatively — since the raw vec is being appended to — sort the
  full raw view once at `Done` and update `view` in-place with binary
  insertion during the load.
- **Predicted improvement**: `open_and_load_10000` drops from ~730 ms
  to under 200 ms (**≥ 3.5×**).
- **Confidence**: high — the bench pattern is unambiguous.

### 2. `atlas_fs::Entry::extension` — allocates `String` per call

- **Location**: `crates/atlas-fs/src/entry.rs:127-133`.
- **Category**: allocation churn in a hot sort comparator.
- **Measured impact**: `sort_10k_extension` = 11.80 ms vs.
  `sort_10k_name_natural_ci` = 3.85 ms; the extension comparator is
  ~3× slower on the exact same input set. Every comparison calls
  `Entry::extension()` twice (once per side), which allocates a
  `String` and lowercases; sorting 10k entries makes ~10k×log2(10k) ≈
  132 k comparisons and therefore ~264 k allocations per sort.
- **Fix hypothesis**: `Entry::extension()` returns a `Cow<str>` (or a
  `&str` slice into `name`) and the sort comparator handles ASCII
  case-folding on the fly. Since names already live in `String`, we
  can slice.
- **Predicted improvement**: `sort_10k_extension` closer to
  `sort_10k_name_natural_ci` (~4-5 ms) — **≥ 2×**.
- **Confidence**: high.

### 3. `atlas_fs::sort::compare_names` — case-insensitive lexi branch allocates per compare

- **Location**: `crates/atlas-fs/src/sort.rs:112-120` (the
  non-natural branch: `a.to_ascii_lowercase().cmp(&b.to_ascii_lowercase())`).
- **Category**: allocation churn (per-compare `String`).
- **Measured impact**: `sort_10k_name_lexi_ci` = 5.97 ms vs.
  natural-CI 3.85 ms. The natural path already iterates bytes and
  does per-byte case-folding; the lexi CI path allocates two `String`s
  per comparison.
- **Fix hypothesis**: fold on the fly with byte iterators (mirror of
  the natural path).
- **Predicted improvement**: `sort_10k_name_lexi_ci` drops to ≈ 2 ms
  (**≥ 2.5×**).
- **Confidence**: high.

### 4. `atlas_search::fuzzy::fuzzy_rank` — matcher created per candidate

- **Location**: `crates/atlas-search/src/fuzzy.rs:22-34` and
  `fuzzy_score` at 10-19.
- **Category**: repeated setup — every candidate rebuilds the
  `nucleo::Matcher` and re-parses the pattern.
- **Measured impact**: `rank_10k_candidates_shell` = 22.0 ms — plenty
  of headroom above the MVP goal (1M docs in 50 ms), but the palette
  currently ranks a few thousand items per keystroke, so any per-call
  overhead compounds.
- **Fix hypothesis**: build the pattern once and reuse a single
  `Matcher` across all candidates.
- **Predicted improvement**: 2–4× on 10k, since nucleo's matcher is
  meant to be reused.
- **Confidence**: high.

### 5. `atlas_fs::filter::CompiledFilter::matches` — allocates per entry when a substring query is present

- **Location**: `crates/atlas-fs/src/filter.rs:100-104`.
- **Category**: allocation churn.
- **Measured impact**: `matches_substring_10k` = 466 µs vs.
  `matches_no_query_10k` = 24 µs → an order of magnitude in the
  substring case. For the pane-level "type-to-filter" affordance
  this fires on every character.
- **Fix hypothesis**: case-insensitive substring search without
  allocating — `name.as_bytes().windows(q.len()).any(...)` with
  per-byte fold, or use `memchr::memmem` with a case-folded query
  and a case-folded slice cursor. Simpler still: leverage
  `std::str::find` on a case-folded scan (short-circuits fast).
- **Predicted improvement**: `matches_substring_10k` under 60 µs
  (**≥ 5×**).
- **Confidence**: high.

### 6. `atlas_ops::primitives::copy::add_bytes_progress` — clones `PathBuf` per 1 MiB

- **Location**: `crates/atlas-ops/src/primitives/copy.rs:513-521`.
- **Category**: allocation churn on I/O hot path; unnecessary UI event volume.
- **Measured impact**: not visible in the two current copy benches
  because both fit easily in a single 1 MiB buffer read; but for a
  1 GB file the current code performs 1 024 `PathBuf::to_path_buf`
  clones and 1 024 mutex-locked writes, each of which also races
  with the queue's status thread reading `progress.current_path`.
- **Fix hypothesis**: only update `current_path` on file transitions
  (in `update_current_path`, already called before the byte loop
  starts); `add_bytes_progress` only needs to bump the counter.
- **Predicted improvement**: subtle for the 16 MiB bench; big for
  multi-GB files. Cold-path win with obvious correctness — mark as
  "cold path" and land it.
- **Confidence**: medium (measurable win at large sizes only).

### 7. `atlas_thumbs::decode::can_thumbnail` — allocates a `String` per extension check

- **Location**: `crates/atlas-thumbs/src/decode.rs:19-28` and 34-38.
- **Category**: allocation churn on a call that is invoked for every
  entry when a grid/gallery view repopulates thumbnails.
- **Measured impact**: not benched (cold path — never sees more than
  the visible entries after virtualization), but the fix is trivial
  and consistent with the "no per-item allocation" principle.
- **Fix hypothesis**: use `eq_ignore_ascii_case` on the extension
  slice; do not lowercase.
- **Confidence**: high — cold-path cleanup, no measurement needed.

### 8. `atlas_fs::view_model` snapshot/subscription allocation

- **Location**: `view_model.rs:477-479` (`entries()` clones the full
  `Vec<Entry>` per call), `269` (`event.clone()` per subscriber per
  notify).
- **Category**: allocation and copy.
- **Measured impact**: not directly benched — `entries()` is called
  by the shell on every project. Deferring: the fix touches the
  public trait signature (`fn entries(&self) -> Vec<Entry>`), which
  is called from many places. Flag as follow-up.
- **Confidence**: medium (correct win, but scope is larger).

### 9. `atlas_fs::view_model::handle_removed` / `handle_modified` — linear scans

- **Location**: `view_model.rs:346-364` and 372-438.
- **Category**: `O(N)` per fs event.
- **Measured impact**: not directly benched; on directories with tens
  of thousands of entries during frantic file activity (e.g. a build
  running) this becomes a scaling wall. Real fix requires a
  name-indexed side-table.
- **Confidence**: medium — flag as follow-up.
