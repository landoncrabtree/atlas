## Optimizations by commit

Each entry lists the file(s) touched, the specific principles from
`.github/instructions/performance.instructions.md` it addresses, and
the measurement rationale.

### 22e3396 — perf(atlas-fs): make view-model loader linear, not quadratic

- Files: `crates/atlas-fs/src/view_model.rs`
- Principle: #2 (streaming over batching, without a quadratic
  accumulator), anti-pattern "Allocating per item in hot loops".
- Rationale: `open_and_load_10000` was 20.8× slower than
  `open_and_load_1000` — clear super-linear behaviour. The
  `Inner::merge_batch` fix keeps the loader linear in total entry
  count.

### 894f9a4 — perf(atlas-fs): stop allocating per-compare Strings in the sort comparators

- Files: `crates/atlas-fs/src/entry.rs`, `crates/atlas-fs/src/sort.rs`
- Principle: anti-patterns "Allocating per item in hot loops" and
  "clone() of large strings to satisfy the borrow checker".
- Rationale: sort of 10k entries did O(N log N) short-lived String
  allocations in the case-insensitive lexi and extension comparators.
  Replaced with `Cow` + on-the-fly byte fold. `sort_10k_name_lexi_ci`
  dropped by 71%.

### fde6723 — perf(atlas-search): reuse nucleo Matcher + Pattern across fuzzy_rank candidates

- Files: `crates/atlas-search/src/fuzzy.rs`
- Principle: #5 (measured before optimizing), anti-pattern
  "Allocating per item in hot loops".
- Rationale: `fuzzy_rank` created a fresh `Matcher` per candidate
  even though nucleo's Matcher is explicitly designed to be reused
  (owns a scoring buffer). 28.8× speedup on 10k candidates.

### d4c26a6 — perf(atlas-fs): allocation-free case-insensitive substring filter

- Files: `crates/atlas-fs/src/filter.rs`
- Principle: anti-pattern "Allocating per item in hot loops".
- Rationale: `matches_substring_10k` dominated by one
  `to_ascii_lowercase()` per candidate. Replaced with inline
  byte-level fold, using the same shape as the already-alloc-free
  natural-sort comparator. 6.8× speedup.

### 5ee222b — perf(atlas-ops): stop cloning source PathBuf on every 1 MiB copy iteration

- Files: `crates/atlas-ops/src/primitives/copy.rs`
- Principle: #6 (cross-thread coordination is explicit — do not
  thrash the shared mutex), anti-pattern "Allocating per item in
  hot loops".
- Rationale: `add_bytes_progress` re-set `progress.current_path`
  on every 1 MiB read even though the path is invariant across the
  loop and was already set by `update_current_path`. Trivially
  correct removal. Small measurable wins on small-file bench,
  meaningful on multi-GB files (bench does not cover).

### d3a944c — perf(atlas-thumbs): use eq_ignore_ascii_case for can_thumbnail extension check

- Files: `crates/atlas-thumbs/src/decode.rs`
- Principle: anti-pattern "Allocating per item in hot loops".
- Cold-path cleanup — no measurement. Consistent with the audit's
  overall pattern; the previous form allocated a lowercase copy per
  entry the grid / gallery views considered for thumbnailing.

### d99ee19 — perf(atlas-ui): allocation-free case-insensitive path match in the goto palette

- Files: `crates/atlas-ui/src/palette/source.rs`
- Principle: anti-pattern "Allocating per item in hot loops".
- Cold-path cleanup — the palette matcher is not directly benched
  today (follow-up), but the fix follows the same shape as the
  filter change above (`contains_ascii_ci` helper) and mechanically
  eliminates up to 2×`MAX_CACHED_PATHS` (200k) allocations per
  keystroke on a warm cache.
